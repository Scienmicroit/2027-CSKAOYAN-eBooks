**15.** 以下是一段关于进程调度与同步的论述：

"在操作系统中，当一个正在运行的进程因等待 I/O 操作而 ① 时，该进程会 ② 调用阻塞原语，从 ③ 态转换为 ④ 态。该转换是进程的 ⑤ 行为。当 I/O 操作完成后，由 ⑥ 将进程状态改为就绪态，该转换是 ⑦ 行为。需要注意的是，若系统在进程处于 ⑧ 态时进行了上下文切换，则必须保存该进程的 CPU 上下文到其 PCB 中。"

- A. ①无法继续执行　②主动　③运行　④阻塞　⑤主动　⑥中断处理程序　⑦被动　⑧运行

- D. ①无法继续执行　②主动　③运行　④阻塞　⑤被动　⑥调度程序　⑦主动　⑧就绪

## 答案与解析

---

**01. D（全部正确）**

逐一验证：
- **I ✅** 运行态→阻塞态是进程的主动行为。教材明确指出："阻塞是进程的主动行为，因此只有处于运行态的进程才能执行阻塞操作。"进程因等待 I/O 或申请资源而主动调用阻塞原语。
- **II ✅** 阻塞态→就绪态是被动行为，由外部事件（如 I/O 完成中断）触发。教材指出："从阻塞态变为就绪态则是一种被动行为，依赖于外部事件的发生或其他进程的协助。"
- **III ✅** 就绪态→运行态由调度程序（内核）完成，进程本身无法决定何时被调度。
- **IV ✅** 阻塞态→运行态不能直接发生。须先转为就绪态，再等待调度选中。教材明确标注此转换"不可能"。

四个说法全部正确，选 D 说明对进程状态转换的主被动关系掌握非常扎实。

---

**02. B（仅 I、II、IV）**

- **I ✅** ULT 切换由线程库在用户空间完成，不涉及内核，无须模式切换，开销小。
- **II ✅** KLT 切换时：同一进程内的线程共享地址空间，无需切换页表；但调度由内核完成，必须执行系统调用从用户态切换到内核态，因此切换开销大于 ULT。
- **III ❌** 高难度陷阱。这是 ULT 的经典致命缺陷——当某个 ULT 执行阻塞型系统调用（如 read()）时，内核认为 **整个进程被阻塞**，因而 **所有 ULT 都被阻塞**。因为内核只能感知到进程的 PCB，无法感知进程内部的多个 ULT。正确说法应为一个 ULT 阻塞→全进程阻塞。
- **IV ✅** KLT 每个线程有独立的内核态 TCB，由内核维护，内核可感知并调度每个线程。

---

**03. D**

- A ✅ 共享存储避免了数据在内核态与用户态之间的拷贝，速度最快；OS 只负责分配和映射地址，不提供自动同步保护。
- B ✅ 无名管道半双工单向通信，数据按 FIFO 字节流读写，读完即释放空间；双向通信需两个管道。
- C ✅ 信号处理时机在进程从内核态返回用户态时，教材明确。
- **D ❌** 细微但关键的陷阱。"直接通信"是指发送方直接在 send 原语中 **指定接收方的身份**，而非直接写入接收方的地址空间。数据仍需 **经内核中转**（复制到内核缓冲队列），再由内核递交给接收方。教材原文："内核负责将消息从发送方复制到接收方的内核缓冲队列中。"D 将"直接通信"偷换为"无内核介入"，属于概念混淆。

---

**04. A（仅 I、III、IV）**

- **I ✅** FCFS：长作业（CPU 密集型）先到有利，短作业（I/O 密集型）排在长作业后等待时间长。
- **II ❌** 关键陷阱。SJF 的"平均等待时间最优"性质有一个重要前提——**所有作业同时到达**。教材明确标注："当所有作业同时到达且运行时间已知时，SJF 算法的平均等待时间和平均周转时间是最优的。"当到达时间不同时，该结论不成立。
- **III ✅** 响应比公式 = (等待时间 + 估计运行时间) / 估计运行时间。长作业的响应比随等待时间增加而提高，最终可被调度，不会饥饿。
- **IV ✅** 时间片足够大时，轮流执行一次即完成，退化为 FCFS。

---

**05. D**

- A ✅ 新进程首先进入最高优先级队列队尾，该队列采用 RR 调度。
- B ✅ 高优先级队列时间片小，低优先级队列时间片大，通常相差 2 倍关系。
- C ✅ 仅当所有高优先级队列为空时才调度低优先级队列。
- **D ❌** 非常精妙的陷阱。最低优先级队列的时间片虽最长，但 **仍然采用 RR（时间片轮转）调度**，不是 FCFS。教材原文："最低优先级队列（如第 n 级）的时间片较长，其行为接近 FCFS，**但仍采用时间片轮转**以保证公平性。""接近 FCFS" ≠ "是 FCFS"。

---

**06. C**

分析修改后的代码：
```
P1 (modified):              P2 (unchanged):
  while(1) {                  while(1) {
    生产数据;                     P(full);
    V(full);  // 提前释放        从缓冲区取数据;
    P(empty); // 后申请           V(empty);
    将数据放入缓冲区;              消费数据;
  }                           }
```

empty = 1, full = 0。

若 P1 先运行：
1. 生产数据 → V(full): full=1 → P(empty): empty=1→0 → 写入数据（第一次，正常）
2. 第二次循环：生产数据 → V(full): full=2（但缓冲区容量为 1，此信号量值已超出实际含义）→ P(empty): empty=-1，P1 阻塞

此时缓冲区已有一个数据。P2 被调度：P(full) 成功（full=2→1），取走数据，V(empty): empty=0。消费。

再次调度 P2：P(full): full=1→0，取数据——但 **缓冲区实际上已空**（唯一数据已被取走），P2 读取到无效数据。

与此同时 P1 因 empty>0 被唤醒，可再次写入前未确保旧数据已被取走。

因此：**P1 可能连续写入多个数据（通过 V(full) 积累信号量），导致缓冲区中的数据被覆盖而未被消费**。C 正确。

- A ❌ 有明显问题。
- B ❌ 并非交替执行，P1 可连续执行多次循环。
- D ❌ 不会死锁——P1 最终会在 P(empty) 上阻塞，此时 P2 可获得 CPU 执行。

**07. A（仅 I、II）**

信号量 S 初值为 3（3 个可用资源），当前 S = -1。

对于记录型信号量：
- 当 S.value > 0 时，表示还有该数量的可用资源
- 当 S.value ≤ 0 时，S.value 的绝对值 = 阻塞等待的进程数，且可用资源为 0

当前 S = -1：
- 所有资源已被分配完毕，可用资源数为 **0**。说法 I ✅
- |S| = 1，等待进程数为 **1**。说法 II ✅
- 从初值 3 到当前值 -1 的过程中，资源分配满 3 个后还多出一个请求被阻塞，因此已获得资源的进程数为 **3**（全部资源被占用），而非 4。说法 III ❌
- 等待进程数为 1，不是 2。说法 IV ❌

**08. A（仅 I、II、III）**

- **I ✅** 资源分配图中有环 ≠ 死锁。教材明确："资源分配图有环 ⇏ 死锁（某类资源多实例时，环不必然死锁）"。只有当每类资源仅一个实例时，环才等价于死锁。
- **II ✅** 死锁 vs 饥饿关键区别：死锁进程均处于阻塞态；饥饿进程可能处于就绪态（如 SPF 中长作业长期得不到 CPU）。教材明确。
- **III ✅** 不可剥夺条件：资源不能强行拿走。请求并保持：已有资源不释放，同时请求新的。
- **IV ❌** 知识点硬伤。不安全状态 **可能** 发生死锁，但 **不一定** 发生。教材明确："不安全状态 ⇏ 一定有死锁"。IV 说"一定不会"完全相反。

**10. C**

- A ✅ 管程封装共享数据和操作，自动保证互斥——任一时刻仅一个进程可执行管程内过程。
- B ✅ 信号量 P、V 分散在各进程，开发者需自行保证顺序正确，容易出错。
- **C ❌** 关键区辨。管程条件变量的 signal 操作 **与信号量的 V 操作有本质区别**：
  - 信号量的 V 操作 **总是执行 S.value++**，无论是否有等待进程。
  - 管程条件变量的 signal 操作 **若没有进程在该条件变量上等待，则不产生任何效果**，不会"增加计数"。
  - 条件变量本身不维护计数值，仅维护等待队列。
- D ✅ 条件变量不计数，仅排队通知；信号量的值反映可用资源数。

**11. A（仅 I、II、III）**

- **I ✅** 模式切换在同进程内完成（用户态↔内核态），不改变当前进程。
- **II ✅** 上下文切换 = 进程切换，涉及 PCB 的保存与恢复。
- **III ✅** 模式切换不切换进程；但进程切换一定发生在内核态（须先通过中断/异常/系统调用进入内核态，才能执行调度和切换），因此必然伴随模式切换。
- **IV ❌** 细微错误。模式切换中，用户态→内核态的部分由 **硬件**（中断/异常/系统调用的响应机制）自动完成，并非全程由内核完成。硬件负责保存断点（PC、PSW）和切换 CPU 模式，此后内核才接管。说"只能由内核完成"不准确。

**12. B（仅 II、IV）**

- I ❌ 统考经典易错点。处于临界区的进程同样 **可以被调度**。教材："处于临界区的进程也可能因中断或抢占而导致调度。此外，若进程在临界区内请求的是一个需要等待的资源，比如打印机，则它主动放弃 CPU，让其他进程运行。"
- **II ✅** 创建新进程后，父子进程都处于就绪态，可立即进行 CPU 调度。
- III ❌ 临界区只保证互斥访问，不保证不可中断。进程在临界区内仍可能被时钟中断打断。
- **IV ✅** 进程切换时需要保存/恢复程序计数器（PC），而页表基址寄存器存放当前进程页表的物理地址，切换进程时也必须更新，以激活新进程的地址空间映射。

**14. A**

多对一线程模型（ULT）中，内核仅以进程为单位进行调度。系统采用 RR（时间片 = 100ms）：
- 进程 A 获 100ms 时间片，其唯一的 ULT 独占这 100ms。
- 进程 B 获 100ms 时间片，其 50 个 ULT 需 **共享这 100ms**。若进程 B 内部采用公平轮转调度，每个 ULT 平均获得约 2ms。

因此，进程 A 的线程获得的 CPU 时间是进程 B 中各线程的 **50 倍**。A 正确。

- B ❌ 相同？显然不是，50 倍差距。
- C ❌ "每个线程每次获得 2ms" 的表述有歧义——这 2ms 是 **在 100ms 的时间片内** 通过内部调度轮流获得的时间，而非操作系统分配的独立时间片。在 ULT 中，操作系统仍然认为进程 B 获得了 100ms，内部 2ms 是由线程库自行分配的，不是内核分配的时间片。
- D ❌ 50 个线程共享 100ms，不是每个 100ms。
